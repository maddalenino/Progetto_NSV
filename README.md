# Counter web + cache (Docker Compose multi-container)**Autore:** Massimiliano Mariconda  **Codice variante:** A1  **Repo:** https://github.com/maddalenino/Progetto\_NSV\---# 1\. ObiettivoL'obiettivo del progetto è comprendere il funzionamento di Docker Compose e la comunicazione tra container tramite rete privata interna.Il sistema realizzato consiste in una web application Flask che comunica con un database Redis per mantenere un contatore persistente delle visite. Solo la web app è esposta verso l'host, mentre Redis rimane accessibile esclusivamente all'interno della rete Docker Compose.\---# 2\. Architettura```textBrowser Host     |     | localhost:5000     v+-------------------+|   Container web   || Flask + Python    |+-------------------+          |          | rete Docker Compose          | hostname: redis          v+-------------------+| Container Redis   || porta interna     || 6379              |+-------------------+```La web app comunica con Redis usando il nome del servizio `redis`, risolto automaticamente dal DNS interno di Docker Compose.Redis utilizza un volume persistente Docker (`redis-data`) per mantenere il counter anche dopo il riavvio dei container.\---# 3\. PrerequisitiSistema utilizzato:* Windows 11* WSL2 Ubuntu 24.04* Docker Desktop con integrazione WSL2* Docker Engine 26+* Docker Compose v2Verifica installazione:```bashdocker --versiondocker compose version```\---# 4\. Come riprodurre il progetto## Clonare il repository```bashgit clone https://github.com/maddalenino/Progetto\\\_NSVcd progetto```## Rendere eseguibili gli script```bashchmod +x scripts/setup.shchmod +x scripts/teardown.sh```## Avviare i container```bash./scripts/setup.sh```Output atteso:* build dell'immagine Flask* download immagine Redis* avvio di due container:  * counter-web  * counter-redis## Verificare i container```bashdocker compose ps```La porta `5000` deve risultare esposta solo dal container `web`.\---# 5\. Verifica del funzionamento## Test del counterEseguire più volte:```bashcurl http://localhost:5000```Output atteso:```json{"Ciao! Questa pagina è stata visitata 1 volte"}```poi:```json{"Ciao! Questa pagina è stata visitata 2 volte"}```e così via.\---## Redis NON raggiungibile dall'host```bashcurl http://localhost:6379```Output atteso:```textConnection refused```Questo dimostra che Redis non espone porte pubbliche sull'host.\---## Verifica rete interna ComposeEntrare nel container web:```bashdocker compose exec web sh```Test:```bashredis-cli -h redis ping```Output:```textPONG```Questo dimostra che il servizio Redis è raggiungibile internamente tramite DNS Docker.\---## Verifica persistenzaArrestare i container:```bashdocker compose down```Riavviare:```bashdocker compose up -d```Ripetere:```bashcurl http://localhost:5000```Il counter NON riparte da zero.\---## Eliminazione completa del volume```bashdocker compose down -v```Il volume Redis viene eliminato.Al riavvio il counter torna da zero.\---# 6\. RiflessioniQuesto progetto ha mostrato concretamente la differenza tra:* porte interne ai container* porte pubblicate sull'hostRedis infatti rimane completamente isolato dalla macchina host, aumentando la sicurezza dell'applicazione.Docker Compose crea automaticamente una rete privata dove i container possono comunicare tramite il nome del servizio. Questo elimina la necessità di conoscere gli indirizzi IP.L'utilizzo di un volume Docker permette di separare il ciclo di vita dei dati da quello dei container. Anche se i container vengono rimossi, i dati possono sopravvivere.Se Redis fosse esposto con:```yamlports:  - "6379:6379"```chiunque sulla rete host potrebbe collegarsi al database/cache senza autenticazione, creando gravi problemi di sicurezza.\---# 7\. Punti aperti1\) Se venisse pubblicato Redis con ports: \["6379:6379"], Redis diventerebbe raggiungibile anche dall’host tramite localhost:6379.Attualmente invece è accessibile solo internamente tramite la rete privata Docker Compose. Esporlo aumenterebbe la superficie d’attacco perché Redis non è pensato per essere pubblico: un utente potrebbe leggere, modificare o cancellare dati.Nel modo originale invece sonoesposti solo i servizi necessari e si mantengono privati quelli interni all’architettura.2\) Se si scala la web app, le repliche Flask possono aumentare senza perdere coerenza perché non tengono lo stato in memoria: il contatoresta in Redis, che è condiviso e aggiorna il valore in modo atomico. Il vero problema non è il counter, ma la porta pubblicata su host, perché tre repliche non possono tutte usare la stessa 5000 sull’host senza un load balancer.3\) Usando dict in memoria con lo scale si avrebbero stati diversi su container diversi; con il restart si perderebbero completamente idati; con richieste concorrenti si avrebbe più rischio di inconsistenze. Redis risolve proprio questi problemi perché mette lo stato fuori dalla web app, in un servizio condiviso e persistente.4\) Nel cloud il volume Docker locale non è la soluzione giusta per i dati di Redis: al suo posto si potrebbe usare un Redis gestito, comeElastiCache, Memorystore o Azure Cache for Redis. EFS servirebbe invece solo se fosse necessario condividere file tra container, perchéè un file system, non un database Redis.# 8\.
